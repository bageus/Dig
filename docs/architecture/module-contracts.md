# Контракты и зависимости модулей Dig

## 1. Назначение

Документ описывает публичные контракты, события, правила зависимостей и минимальный вертикальный срез. Структура директорий находится в `module-structure.md`.

## 2. Публичные контракты

Каждый крупный модуль предоставляет минимальный фасад. Внутренние коллекции и изменяемые сущности наружу не выдаются.

Пример World:

```text
IWorldService
  GetCell(cellId)
  ValidateDesignation(command)
  ApplyTerrainChange(change)
  CreateSnapshot(region)
```

Пример Jobs:

```text
IJobService
  CreateWorkOrder(spec)
  FindCandidates(agentId)
  ClaimJob(agentId, jobId)
  AdvanceJob(jobId, result)
  CancelJob(jobId, reason)
```

Пример Inventory:

```text
IInventoryService
  QueryAvailableItems(filter)
  ReserveItems(request)
  Transfer(transfer)
  ReleaseReservation(id)
```

Публичный контракт должен:

- выражать доменные операции;
- не раскрывать детали хранения;
- не возвращать изменяемые внутренние коллекции;
- использовать стабильные идентификаторы;
- отделять команды от запросов;
- возвращать проверяемый результат или ошибку.

## 3. События между системами

Примеры фактов:

```text
CellChanged
ChunkInvalidated
JobCreated
JobClaimed
JobCompleted
ReservationReleased
ItemTransferred
BuildingCompleted
TechnologyUnlocked
AgentNeedBecameCritical
AgentDied
ChildBorn
FactionRelationChanged
```

Событие содержит идентификаторы и минимально необходимый снимок данных. Получатель не изменяет состояние отправителя через ссылку на объект.

Событие не должно:

- быть скрытой командой;
- требовать обязательного порядка неизвестных подписчиков;
- переносить большие изменяемые графы объектов;
- становиться вторым журналом авторитетного состояния.

## 4. Внутренняя структура системы

По мере необходимости система может использовать шаблон:

```text
<System>/
  Model/
  Commands/
  Events/
  Queries/
  Policies/
  Services/
  Serialization/
  Tests/
```

Пустые директории заранее не создаются. Структура появляется только вместе с реальной ответственностью.

## 5. Правила разбиения файлов

При достижении 250–300 строк следует проверить, не смешаны ли:

- модель и оркестрация;
- валидация и выполнение;
- сериализация и доменная логика;
- выбор стратегии и реализация стратегии;
- публичный контракт и детали движка;
- разные состояния конечного автомата;
- код и большой декларативный набор данных.

Файл более 350 строк считается нарушением, кроме документированных исключений из `docs/development-rules.md`.

Разделение выполняется по ответственности. Запрещено создавать `Part1`, `Part2` или аналогичные файлы без смысловых границ.

## 6. Разрешённые зависимости

```text
Presentation -> Application
Application -> Domain
Infrastructure -> Domain interfaces
Infrastructure -> Application interfaces
Content loader -> Content schemas
Diagnostics -> public read-only contracts
Tests -> public and selected internal test contracts
```

## 7. Запрещённые зависимости

```text
Domain -> Presentation
Domain -> Engine API
Domain -> File system
World -> UI
Jobs -> concrete animation controller
Inventory -> scene object
Content definitions -> mutable runtime entity
Navigation -> terrain mesh as source of truth
Saving -> ownership of live domain state
```

## 8. Взаимодействие без циклов

Если двум модулям нужны данные друг друга, используется один из вариантов:

1. общий минимальный контракт в более внутреннем слое;
2. доменное событие;
3. read-only query interface;
4. Application orchestration;
5. snapshot с версией.

Прямая двусторонняя ссылка модулей запрещена.

## 9. Версионирование производных данных

Маршруты, меши, spatial indexes и другие кэши должны содержать версию исходного snapshot.

Результат фоновой задачи применяется только если:

- сущность ещё существует;
- версия источника совпадает;
- запрос не отменён;
- результат прошёл валидацию.

## 10. Минимальный вертикальный срез

Первый сквозной сценарий:

1. пользователь назначает одну клетку на копание;
2. Application создаёт команду;
3. World валидирует назначение;
4. Jobs создаёт работу;
5. Agent выбирает и резервирует её;
6. Navigation возвращает маршрут;
7. Agent выполняет действие;
8. World меняет клетку;
9. Inventory создаёт ресурс;
10. Presentation перестраивает затронутый чанк;
11. состояние сохраняется и загружается;
12. сценарий проверяется интеграционным тестом.

## 11. Критерии готовности модуля

Модуль считается готовым к использованию, когда:

- определён владелец состояния;
- существует минимальный публичный контракт;
- команды и запросы разделены;
- опубликованы необходимые события;
- описаны инварианты;
- отсутствуют циклические зависимости;
- есть тесты успешных и ошибочных сценариев;
- есть диагностический read model для неочевидного поведения;
- определён формат сохранения или явно указано, что состояние производное;
- ручные файлы укладываются в 350 строк.
