# Верхнеуровневая архитектура Dig

## 1. Цель

Этот документ фиксирует общую структуру проекта Dig и направления зависимостей между крупными подсистемами. Детали модулей находятся в `module-structure.md`, каталог систем — в `system-catalog.md`, обязательные правила разработки — в `../development-rules.md`.

## 2. Архитектурная модель

```text
┌─────────────────────────────────────────────────────────────┐
│ Presentation                                                │
│ Camera · UI · Input · Animation · Audio · VFX · Debug Views │
└──────────────────────────────┬──────────────────────────────┘
                               │ Commands / Queries
┌──────────────────────────────▼──────────────────────────────┐
│ Application                                                 │
│ Game modes · Use cases · Command routing · Orchestration    │
└──────────────────────────────┬──────────────────────────────┘
                               │ Domain contracts
┌──────────────────────────────▼──────────────────────────────┐
│ Simulation / Domain                                         │
│ World · Agents · Jobs · Economy · Society · Combat · Time   │
└───────────────┬──────────────────────────────┬──────────────┘
                │ Definitions                  │ Ports
┌───────────────▼──────────────┐  ┌────────────▼──────────────┐
│ Content / Data              │  │ Infrastructure            │
│ Materials · Items · Recipes │  │ Save · Load · Files · Log │
│ Buildings · Tech · Balance  │  │ Engine adapters · Threads │
└─────────────────────────────┘  └───────────────────────────┘
```

## 3. Главный принцип владения состоянием

Каждая подсистема владеет только своим состоянием.

```text
World          -> клетки, чанки, среды, исследованность
Navigation     -> производный граф проходимости и маршруты
Agents         -> идентичность и runtime-состояние жителей
Needs          -> значения потребностей и модификаторы
Jobs           -> жизненный цикл работ и резервирования
Inventory      -> местонахождение и количество предметов
Buildings      -> строительство и runtime-состояние объектов
Production     -> очереди и прогресс производственных заказов
Technology     -> открытия и доступность контента
Society        -> отношения, семьи, поколения
Combat         -> боевые намерения, урон и статусы
Factions       -> дипломатия и стратегические цели
Saving         -> снимок авторитетного состояния и миграции
Presentation   -> только визуальная проекция
```

Ни один визуальный объект не является источником истины.

## 4. Главный поток игры

```text
Input
  -> Application Command
  -> Domain validation
  -> State mutation
  -> Domain Events
  -> Dependent systems react
  -> Read models / snapshots
  -> Presentation update
```

Пример копания:

```text
Игрок отмечает клетки
  -> CreateDigDesignationCommand
  -> World проверяет возможность назначения
  -> Jobs создаёт работы копания
  -> Worker резервирует работу и инструмент
  -> Navigation строит маршрут
  -> Worker выполняет этапы
  -> World изменяет клетку
  -> Chunk помечается dirty
  -> Navigation инвалидирует затронутую область
  -> Inventory создаёт добытый ресурс
  -> Presentation обновляет меш и эффекты
```

## 5. Основные bounded contexts

### 5.1. World Simulation

Отвечает за логическую карту, клетки, чанки, вещества, жидкости, температуру, опасности и изменения среды.

### 5.2. Agent Simulation

Отвечает за жителей, потребности, навыки, здоровье, расписания, намерения, жизненный цикл и личные приоритеты.

### 5.3. Work Management

Отвечает за создание работ, оценку приоритетов, назначение исполнителей, резервирование ресурсов, выполнение этапов и отмену.

### 5.4. Colony Economy

Отвечает за предметы, склады, транспортировку, здания, рецепты, производство, энергию и доступность ресурсов.

### 5.5. Society Simulation

Отвечает за отношения, партнёрства, семьи, детей, поколения, социальные активности и долгосрочное развитие поселения.

### 5.6. Conflict Simulation

Отвечает за угрозы, оружие, бой, лечение, кланы, дипломатию и стратегический ИИ противников.

### 5.7. Runtime Platform

Отвечает за игровой цикл, сохранения, загрузку контента, фоновые вычисления, диагностику и адаптацию к движку.

## 6. Частоты обновления

Системы не обновляются одинаково часто.

```text
Каждый кадр:
- камера;
- интерполяция визуального движения;
- анимации;
- UI.

Фиксированный simulation tick:
- перемещение;
- выполнение текущего действия;
- локальные взаимодействия;
- бой.

Редкий tick или события:
- потребности;
- выбор новой работы;
- отношения;
- производство;
- стратегический ИИ;
- симуляция дальних областей.
```

Конкретные частоты являются настройками симуляции и не зависят от FPS.

## 7. Детерминизм

Доменная симуляция должна быть максимально воспроизводимой.

Для воспроизведения используются:

- seed мира;
- версия генератора;
- фиксированный simulation tick;
- контролируемые random streams по подсистемам;
- стабильные идентификаторы сущностей;
- журнал значимых команд для отладки.

Визуальные эффекты могут быть недетерминированными, если не влияют на игровое состояние.

## 8. Производные представления

Разрешённые производные данные:

- меши чанков;
- коллайдеры;
- навигационные графы;
- spatial indexes;
- UI read models;
- кэши доступности ресурсов;
- агрегаты для далёкой симуляции.

У каждого производного представления должны быть версия источника, dirty-флаг или иной механизм инвалидирования и возможность полной перестройки.

## 9. Многопоточность

Параллельно могут выполняться только вычисления, не нарушающие владение состоянием.

Предпочтительная модель:

1. основной поток создаёт неизменяемый snapshot входных данных;
2. worker вычисляет маршрут, меш, генерацию или оценку;
3. результат возвращается с версией источника;
4. основной поток применяет результат, только если версия актуальна.

Прямое конкурентное изменение доменных коллекций запрещено.

## 10. Сохранения

Сохранение содержит авторитетное состояние, но не обязано хранить пересчитываемые кэши.

Снимок включает:

- версию формата;
- параметры и версию генератора;
- изменения мира;
- сущности и их стабильные идентификаторы;
- инвентари;
- активные работы и производственные заказы;
- технологии;
- отношения и семьи;
- состояние фракций;
- время и random state, если он необходим для воспроизводимости.

## 11. Независимость от движка

Адаптер движка реализует:

- создание визуальных объектов;
- проигрывание анимаций и звуков;
- чтение ввода;
- работу с потоками;
- доступ к файлам;
- платформенные функции.

Доменный код не должен содержать типов Unity, Godot или другого движка.

## 12. Архитектурные критерии готовности

Верхнеуровневая архитектура считается реализованной, когда:

- модули имеют явные публичные контракты;
- зависимости направлены внутрь к Domain;
- каждая сущность имеет одного владельца состояния;
- команды, события и запросы разделены;
- симуляция запускается без Presentation;
- базовый сценарий можно проверить автоматическим тестом;
- визуальный слой можно пересоздать из доменного состояния;
- сохранение не зависит от объектов сцены;
- диагностические представления объясняют работу ключевых систем.
