# ADR-0002: Save v4 для прогрессии навыков жителей

- Статус: accepted
- Дата: 2026-07-21
- Владельцы: команда Dig

## Контекст

Issues #103–#107 вводят авторитетное состояние из 12 навыков, общий предел,
ключи идемпотентности и последний отчёт перераспределения. Эти данные нельзя
восстановить из World, Jobs или UI. Формат v3 их не содержит, а изменение
precision должно выполняться детерминированно и только один раз.

## Варианты

### Вариант A: сохранить навыки внутри существующих DTO жителя

Плюс — меньше отдельных секций. Минус — текущий вертикальный срез не сохраняет
полный aggregate жителей; это смешало бы lifecycle жителя и прогрессию и
потребовало бы несвязанного расширения формата.

### Вариант B: отдельная версия и секция `AgentSkills`

Плюс — явное владение, независимая валидация, детерминированный порядок и
последовательная миграция. Минус — после загрузки секцию нужно применить к
существующему `IAgentRepository` отдельным restorer.

## Решение

Выбран вариант B. `SaveFormat.CurrentVersion` повышен до 4. Документ содержит
отдельную секцию `AgentSkills`, отсортированную по стабильному `AgentId` и
`AgentSkillId`. Сохраняются schema/precision/units-per-point, capacity, все 12
значений, applied source keys, последний redistribution report и migration steps.

Миграция v3→v4 создаёт пустую секцию и не выдумывает значения для отсутствующих
в v3 жителей. Legacy runtime capabilities преобразуются чистым snapshot-query.
Precision v0→v1 масштабируется целыми числами с largest-remainder и фиксируется
в `SaveMigrationReport`; устаревший redistribution report удаляется, потому что
его единицы относятся к старой precision.

## Последствия

### Положительные

- сохранение остаётся координатором snapshot, а не владельцем live-state;
- повтор result после load остаётся no-op благодаря applied source keys;
- migration детерминирована и диагностируема;
- сохранение legacy resident не изменяет aggregate и его `Version`.

### Отрицательные

- composition root обязан вызвать `LoadedAgentSkillProgressionRestorer` после
  восстановления жителей;
- precision migration должна поддерживаться при каждом будущем изменении units.

### Нейтральные и последующие действия

- когда полный lifecycle жителей войдёт в save, секция может быть вложена в его
  DTO отдельным ADR и последовательной версией формата;
- fixture `save-v3.json` сохраняется неизменяемым.

## Критерии пересмотра

Решение пересматривается при появлении полного authoritative resident save,
смене precision, каталога навыков или модели idempotency keys.
