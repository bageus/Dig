# Правила разработки Dig

## 1. Назначение

Этот документ является единственным источником правил разработки проекта Dig. Архитектурные документы описывают устройство игры, но не должны дублировать или переопределять перечисленные здесь правила.

## 2. Один источник истины

Для каждого типа состояния должен существовать ровно один авторитетный владелец.

Примеры:

- состояние клетки хранится в модели мира, а не одновременно в меше, коллайдере и навигации;
- количество предметов хранится в инвентаре, а UI только отображает его;
- состояние жителя хранится в симуляции, а анимация является представлением;
- доступность технологии определяется технологической системой, а не кнопкой интерфейса;
- статус задания определяется системой работ, а не самим работником или производственным зданием.

Производные данные разрешены только при соблюдении всех условий:

1. указан исходный авторитетный источник;
2. данные можно полностью пересчитать;
3. существует явный механизм инвалидирования;
4. производная копия не изменяется напрямую;
5. рассинхронизация обнаруживается диагностикой или тестами.

Запрещено создавать несколько изменяемых копий одного состояния ради удобства.

## 3. Системная логика

Любая новая механика сначала описывается как система:

- какое состояние ей принадлежит;
- какие команды она принимает;
- какие события публикует;
- какие инварианты поддерживает;
- от каких систем зависит;
- кто имеет право изменять её данные;
- как она сохраняется;
- как тестируется;
- как диагностируется в игре.

Логику нельзя распределять случайным образом между UI, анимациями, сценами, объектами движка и обработчиками ввода.

## 4. Архитектурные границы

Используются слои:

```text
Presentation -> Application -> Domain
Infrastructure -> Application / Domain через интерфейсы
Data -> Domain через валидированные определения
```

Правила зависимостей:

- Domain не зависит от игрового движка;
- Domain не обращается к UI, сценам, файловой системе и сетевым API;
- Presentation не изменяет доменное состояние напрямую;
- Application преобразует ввод в команды и запросы;
- Infrastructure реализует сохранение, загрузку, логирование и адаптеры;
- межмодульное взаимодействие выполняется через публичные контракты;
- глобальные изменяемые переменные запрещены;
- циклические зависимости между модулями запрещены.

## 5. Ограничение размера файлов

Обычный исходный файл не должен превышать **350 строк**, включая комментарии и пустые строки.

При приближении к лимиту файл разделяется по ответственности, а не механически по размеру.

Допустимые исключения:

- автоматически сгенерированный код;
- миграции данных;
- небольшие декларативные каталоги, которые невозможно разумно разделить;
- сторонний код, находящийся вне основного исходного дерева.

Каждое исключение должно:

- находиться в явно обозначенной директории;
- содержать пометку о происхождении;
- не использоваться как оправдание для больших ручных классов;
- по возможности проверяться отдельным анализатором.

Рекомендуемые внутренние ориентиры:

- класс или структура: одна ответственность;
- метод: обычно до 30–40 строк;
- количество параметров метода: обычно не более пяти;
- публичный интерфейс модуля: минимальный;
- вложенность условий: не более трёх уровней без выделения метода или стратегии.

## 6. Команды, события и запросы

Используются три разных понятия:

- **Command** — намерение изменить состояние;
- **Event** — уже произошедший факт;
- **Query** — чтение без изменения состояния.

Команда должна проходить валидацию владельцем состояния. Событие не должно использоваться как скрытая команда. Запрос не должен иметь побочных эффектов.

## 7. Инварианты

Критические правила оформляются как проверяемые инварианты.

Минимальный набор:

- предмет не может одновременно находиться в двух инвентарях;
- одна единица ресурса не может быть зарезервирована двумя работами;
- завершённая или отменённая работа не удерживает резервирования;
- житель выполняет не более одной активной работы;
- непроходимая клетка не участвует в актуальном навигационном графе;
- идентификаторы сохранённых сущностей уникальны;
- удалённая сущность не остаётся в индексах и кэшах;
- представление не является владельцем игрового состояния.

## 8. Data-driven контент

Определения предметов, материалов, зданий, рецептов, технологий, потребностей и работ хранятся отдельно от runtime-состояния.

Каждое определение имеет стабильный идентификатор. Ссылки по отображаемым именам запрещены.

Загрузка контента включает:

1. синтаксическую проверку;
2. проверку уникальности идентификаторов;
3. проверку ссылок;
4. проверку циклов;
5. проверку обязательных полей;
6. формирование неизменяемого runtime-каталога.

## 9. Наблюдаемость

Каждая автономная система должна объяснять свои решения.

Для жителя должны быть доступны:

- текущее намерение;
- выбранная работа;
- причина выбора;
- отклонённые альтернативы;
- активные резервирования;
- текущий маршрут;
- блокирующая потребность;
- последний переход состояния.

Для мира должны отображаться dirty-чанки, версии навигации, очереди перестроения и стоимость обновлений.

## 10. Тестирование

Обязательные уровни:

- unit-тесты доменных правил;
- property-based тесты генерации и инвариантов;
- интеграционные тесты взаимодействия систем;
- тесты сериализации и миграции сохранений;
- детерминированные симуляционные сценарии;
- performance-тесты на целевом количестве жителей и размере мира.

Исправление ошибки должно по возможности сопровождаться тестом, который воспроизводит её до исправления.

## 11. Изменение архитектуры

Существенное архитектурное решение оформляется ADR в `docs/adr/`.

ADR обязателен для:

- смены модели мира;
- изменения направления зависимостей;
- выбора формата сохранений;
- введения нового глобального механизма обмена сообщениями;
- изменения модели многопоточности;
- замены навигационного подхода;
- введения исключения из правил этого документа.

## 12. Definition of Done

Задача считается завершённой, когда:

- определён владелец состояния;
- соблюдены границы модулей;
- отсутствует второй источник истины;
- ручные файлы не превышают 350 строк;
- добавлены необходимые тесты;
- добавлена диагностика для неочевидного поведения;
- сохранение поддержано либо явно не требуется;
- документация обновлена;
- критерии приёмки GitHub Issue выполнены.
