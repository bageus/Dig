# Коробки зданий, размещение, сборка и упаковка

Статус: целевая проектная спецификация. Реализация отслеживается в #118.

## 1. Назначение

Все размещаемые здания используют единый физический lifecycle коробки. Здание производится как предмет-коробка, размещается игроком в виде призрачного плана, доставляется свободным гномом на площадку, собирается и позднее может быть разобрано обратно в коробку.

Эта модель заменяет для соответствующего `BuildingDefinition` прямую доставку полного списка строительных материалов на площадку. Одновременное применение двух моделей к одному зданию запрещено.

## 2. Владение состоянием

- `InventoryState` владеет коробкой, количеством, местоположением и reservations.
- `BuildingsState` владеет планом, footprint, orientation, progress, completed state и durability.
- `Jobs` владеет pickup, delivery, assembly и packing lifecycle, исполнителем и позициями.
- `Production` создаёт коробку как обычный output рецепта.
- `Navigation` проверяет достижимость коробки, площадки и рабочих позиций.
- `Presentation` владеет только preview, cursor mode, selection и локальным состоянием панели.

Призрачная модель, курсор-здание и подсветка места не являются авторитетным зданием.

## 3. Content model

Каждое размещаемое здание имеет стабильные ссылки:

```text
BuildingDefinition
- BuildingDefinitionId
- BuildingBoxItemId
- PlacementPolicy
- Footprint
- WorkPositions
- AssemblyWork
- PackingWork
- FunctionalCapabilities
```

`BuildingBoxItemId` обозначает обычный предмет Inventory:

- одна коробка представляет одно здание;
- коробки не складываются друг с другом;
- коробка не является вложенным контейнером материалов;
- display name и локализованное название не используются как ID;
- рецепт коробки содержит стоимость производства здания.

## 4. Нижняя контекстная панель

В один момент активен ровно один режим.

### Нет выбранного объекта

Показывается меню копки:

- свободный тоннель;
- шаблоны комнат;
- ластик удаления нарисованных designation и ещё не выполненных участков tunnel/room plans.

Ластик отменяет только выбранные плановые клетки и связанные незавершённые jobs. Уже выкопанный мир не восстанавливается.

### Выбран гном

Показывается его личный инвентарь:

```text
[ Weapon ] [ Main ] [ Cargo ]
```

### Выбрано построенное здание

Показываются функции этого здания: производство, исследования, режимы, хранилище, работники и другие capabilities из definition/runtime snapshot. Справа всегда находится кнопка упаковки, если здание допускает packing.

### Активна коробка или placement tool

Показывается режим размещения с названием здания, ориентацией, причиной валидности/невалидности и отменой preview.

## 5. Запуск placement mode

Режим размещения включается двумя способами.

### Коробка находится в мире

Обычный ЛКМ по доступной собственной коробке включает placement mode. Коробка остаётся в своём `ItemLocation` и не резервируется до подтверждения валидного плана.

### Коробка находится в инвентаре выбранного гнома

ЛКМ по коробке в Inventory включает тот же placement mode. Иконка остаётся в исходной ячейке до успешной Application-команды.

`Alt + ЛКМ` по коробке в мире не включает preview: выбранный гном получает приказ подобрать коробку в личный инвентарь.

## 6. Preview и подтверждение

В placement mode:

- курсор отображает модель здания;
- мир показывает призрачный footprint;
- валидность вычисляется через World, Buildings и Navigation snapshots;
- Presentation не изменяет состояние до команды;
- ЛКМ по валидной позиции создаёт plan;
- клик по невалидной позиции ничего не создаёт и показывает reason code;
- ПКМ отменяет placement mode и не расходует коробку.

Команда подтверждения атомарно:

1. повторно валидирует footprint, orientation и work positions;
2. проверяет существование и доступность выбранной коробки;
3. резервирует коробку за plan;
4. создаёт призрачный `BuildingPlan`;
5. публикует событие создания плана.

Одна коробка не может быть зарезервирована двумя планами.

## 7. Доставка и сборка

После создания плана он попадает в общий список работ.

Свободный подходящий гном:

1. получает job;
2. идёт к зарезервированной коробке;
3. подбирает её через Inventory transaction;
4. несёт к площадке;
5. фиксирует коробку в site location;
6. выполняет assembly work;
7. завершает здание одной подтверждённой операцией.

Если коробка уже находится в инвентаре назначенного гнома, этап внешнего pickup пропускается, но reservation и commit остаются обязательными.

Коробка не существует одновременно как предмет у гнома, предмет на площадке и завершённое здание. При успешном завершении коробка расходуется ровно один раз.

## 8. Отмена и ошибки

До доставки коробки отмена плана освобождает reservation, коробка остаётся на прежнем месте.

После доставки, но до завершения сборки отмена:

- закрывает связанные jobs;
- освобождает worker/position claims;
- возвращает одну коробку в world site location либо другое явно определённое допустимое место;
- не создаёт частично построенное самостоятельное здание;
- не изменяет количество.

Retry не резервирует коробку повторно. Уничтоженная или недоступная коробка переводит plan в диагностируемое blocked/failed состояние.

## 9. Упаковка построенного здания

Кнопка упаковки в панели здания создаёт `PackBuilding` command.

После проверки:

- здание помечается как ожидающее разборки;
- создаётся общий packing job;
- гном приходит к рабочей позиции и выполняет разборку;
- functional places, active orders и reservations завершаются по явной policy;
- после commit здание освобождает footprint;
- в site location создаётся ровно одна коробка соответствующего `BuildingBoxItemId`.

До commit здание остаётся авторитетным объектом и коробка не существует. После commit здание больше не функционирует.

Нельзя упаковать здание, пока это нарушает активную неотменяемую операцию; UI показывает конкретную причину.

## 10. Контекстный ввод

После UI shielding применяется порядок:

1. активный placement mode обрабатывает ЛКМ как подтверждение preview;
2. выбранный Inventory stack обрабатывает targeted drop;
3. `Alt + ЛКМ` по world building box создаёт pickup order;
4. обычный ЛКМ по world building box включает placement mode;
5. ЛКМ по построенному зданию выбирает его и открывает функции;
6. выбранный гном и свободная земля создают move order;
7. без выбранного гнома нижняя панель показывает excavation tools.

Один pointer event создаёт не более одной команды.

## 11. Save/Load

Сохраняются:

- все коробки как обычные Inventory stacks/locations;
- box reservation и owning plan id;
- plan definition, orientation, footprint и progress;
- site commit state;
- active delivery/assembly/packing jobs;
- work/position/item reservations;
- building functional state до начала packing;
- migration/version data.

Preview, cursor и раскрытая панель не входят в authoritative save.

## 12. Диагностика

Inspector показывает:

- BuildingDefinitionId и BuildingBoxItemId;
- source box location;
- plan и reservation owner;
- pickup/delivery/assembly/packing stage;
- выбранного worker;
- footprint/work position validation;
- commit state коробки;
- причины blocked/cancel/failure;
- отчёт сохранения количества.

## 13. Инварианты

- одна коробка имеет одно authoritative location;
- одна коробка принадлежит не более чем одному активному plan;
- preview не резервирует и не расходует предмет;
- plan не создаётся без доступной коробки;
- completed building не хранит скрытый дубликат коробки;
- packing создаёт ровно одну коробку;
- cancel/retry/save-load не теряют и не дублируют коробки;
- один BuildingDefinition не использует одновременно universal box и legacy full-material site flow;
- UI не меняет Buildings или Inventory напрямую.

## 14. Тесты

Обязательны:

- placement из world box и resident inventory;
- invalid footprint без reservation;
- конкурирующие планы за одну коробку;
- доставка другим свободным гномом;
- коробка уже у назначенного гнома;
- cancel до и после site commit;
- missing box, unreachable source и blocked work position;
- сборка ровно один раз;
- упаковка и повторное размещение;
- active building orders при packing;
- save/load на каждой стадии;
- deterministic replay и quantity conservation;
- Unity Play Mode для context panel, preview, shielding и input priority.
