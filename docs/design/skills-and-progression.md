# Навыки гномов и прогрессия за деятельность

## Назначение

Документ фиксирует каталог навыков, источники прогресса, общий предел характеристик и интеграцию с HUD. Навыки принадлежат конкретному гному и повышаются только через подтверждённую симуляцией работу, обучение или боевое действие.

Display name не используется как техническая ссылка. Для логики, сохранений и контента применяются стабильные `AgentSkillId`.

## Общие правила

- Unity-анимация не начисляет опыт и не является источником прогресса;
- отменённая работа не выдаёт полный опыт завершения;
- один и тот же work/action segment не начисляется повторно после retry или load;
- Jobs, Production, Construction, Combat и Services публикуют типизированные факты, но не владеют значениями навыков;
- Skill aggregate применяет grant, capacity rule и дедупликацию одной транзакцией;
- количества опыта, кривые и коэффициенты задаются данными;
- UI читает immutable snapshot и не изменяет навык напрямую.

## Общий предел

Подтверждённая authoritative модель:

- базовый `TotalSkillCapacity` взрослого гнома — `100`;
- университет постепенно расширяет тот же предел до абсолютного максимума `200`;
- все 12 навыков используют один общий pool;
- максимум отдельной характеристики — `100`;
- университет повышает только capacity и не выдаёт скрытый опыт конкретного навыка;
- практическая работа и бой развивают конкретные характеристики;
- UI не показывает параллельную альтернативную capacity.

Значение `120`, ранее указанное в HUD proposal, было ошибочным и больше не используется.

## Поведение заполненного capacity

Подтверждено направление design:

- после заполнения общей суммы рост активной характеристики должен уменьшать другие;
- характеристика с большим текущим значением сильнее подвержена потере;
- операция должна быть детерминированной, объяснимой и атомарной;
- сумма после commit не превышает текущий `TotalSkillCapacity`;
- ни одна характеристика не выходит за диапазон `0..100`;
- один grant формирует один отчёт со всеми gain/loss изменениями;
- Unity random не используется.

Точный алгоритм выбора и величины потерь остаётся открытым в Q-029.

## Результат проверки исходных scripts

Для поиска исторической формулы проверены:

- `scripts/misc/genattribs.tcl`;
- `scripts/classes/zwerg/zwerg.tcl`;
- `scripts/classes/zwerg/z_methods.tcl`;
- `scripts/classes/zwerg/z_procs.tcl`;
- `scripts/classes/zwerg/z_work_common.tcl`;
- `scripts/classes/zwerg/z_work_prod.tcl`;
- `scripts/classes/work/dojo.tcl`;
- `scripts/classes/work/schule.tcl`;
- `scripts/misc/utility.tcl`.

Найдено:

- `atr_ExpMax` хранит общий предел;
- определены семь рабочих и пять отдельных боевых характеристик;
- работа и обучение вызывают встроенную команду `add_expattrib`;
- `prod_anim_exp` делит gain между повторениями работы;
- dojo вызывает `add_expattrib` для конкретного боевого профиля;
- scripts сохраняют и восстанавливают `atr_ExpMax` и значения характеристик.

Не найдено:

- формула уменьшения остальных характеристик;
- weighted selection;
- точная обработка заполненного `atr_ExpMax`.

Следовательно, cap/redistribution находились внутри недоступной встроенной реализации `add_expattrib` или engine layer, а не в проверенных TCL-файлах. Dig не копирует неизвестную реализацию и сохраняет Q-029 открытым до отдельного design-ответа.

## Каменное дело и пещеры

Пороги используют `skill.stonework`:

- средняя пещера — 20;
- большая — 40;
- высокая — 60.

Доступность вычисляется по максимальному текущему Stonework среди допустимых гномов. Падение навыка блокирует только новые размещения; уже подтверждённый plan продолжает выполняться.

## Производство и здания

Recipe/Work definition ссылается на skill profile, а не определяет навык по имени здания или предмета.

Примеры:

- cooking → `skill.cooking`;
- алкоголь/зелья → `skill.alchemy`;
- плавка/литьё → `skill.metallurgy`;
- service cycle → `skill.service`;
- hauling, доставка коробки, сборка и упаковка → `skill.logistics`;
- каменная обработка → `skill.stonework`;
- деревянная обработка → `skill.woodworking`.

Правило одного или нескольких grants для смешанной деятельности остаётся Q-019.

## Боевые навыки

Combat формирует опыт из фактически использованного профиля:

- attack одноручным оружием → `skill.one_handed_combat`;
- attack двуручным оружием → `skill.two_handed_combat`;
- дальнобойная атака → `skill.ranged_combat`;
- безоружная атака → `skill.unarmed_combat`;
- block/defense со щитом → `skill.defense`.

Щит и оружие в одном боевом цикле используют отдельные события; точное правило нескольких grants находится в Q-020.

## Контракт опыта

Рекомендуемый `SkillExperienceGrant` содержит:

- `AgentId`;
- `AgentSkillId`;
- `SourceKind`;
- stable source work/action id;
- `Amount`;
- `Tick`;
- idempotency key;
- diagnostic description.

Точный момент начисления находится в Q-021.

## HUD

Раскрытая строка выбранного гнома показывает пять навыков с наибольшим текущим значением из всех 12:

- сортировка level descending;
- при равенстве stable `AgentSkillId` ascending;
- полный список остаётся в отдельном inspector;
- шкала использует градиент тёмно-синий → зелёный;
- значение и максимум показываются числом/tooltip;
- top-5 и полный inspector читают один snapshot.

## Save/Load и диагностика

Сохраняются:

- значения и XP progress всех 12 skill IDs;
- `TotalSkillCapacity`;
- catalog/schema version;
- idempotency state незавершённых grants, если требуется;
- migration report.

Диагностика показывает current/maximum capacity, сумму уровней, последний источник опыта, отклонённый grant и будущий redistribution report.
