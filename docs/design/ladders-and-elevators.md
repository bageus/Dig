# Лестницы, лифты и персональная мобильность

Связанная задача: #137.

## Исключено и сохранено

В текущей игре отсутствуют вагонетки, рельсы, стрелки и тачки. Для них не создаются runtime definitions и UI.

Сохраняются:

- деревянная, металлическая и кристаллическая лестницы;
- обычный, паровой и кристаллический лифты;
- ездовой хомяк;
- ховерборд.

## Лестницы

Лестница — статическая вертикальная Navigation link с фиксированной длиной:

| Вариант | Длина |
|---|---:|
| Деревянная | 12 клеток |
| Металлическая | 16 клеток |
| Кристаллическая | 24 клетки |

Placement требует непрерывного свободного вертикального span. Лестница соединяет конечные и промежуточные доступные клетки. Упаковка или разрушение немедленно обновляет Navigation. Материалы перевозятся только в личном Inventory.

## Лифт

Лифт имеет шахту, кабину, направление, очередь запросов и логические остановки.

| Вариант | Класс энергии | Вместимость |
|---|---:|---:|
| Обычный | 1 | 4 гнома |
| Паровой | 2 | 4 гнома |
| Кристаллический | 3 | 4 гнома |

Скорость и расход энергии остаются data-driven balance values.

### Запросы и остановки

- гном создаёт request с исходной и целевой высотой;
- автономный маршрут останавливается только там, где к шахте примыкает доступная площадка/тоннель;
- прямое управление может выбрать любую высоту шахты;
- запросы по текущему направлению обслуживаются по пути;
- противоположные запросы ждут разворота;
- порядок детерминирован.

### Посадка

- кабина имеет четыре места;
- лифт ждёт всех гномов, подошедших в радиус одной клетки, пока есть свободные места;
- timeout отсутствует;
- заполненная кабина оставляет остальных в очереди;
- отдельного грузового режима нет;
- груз существует только в personal Inventory пассажира.

### Потеря энергии

Если питание исчезает во время движения:

1. кабина останавливается;
2. пассажиры переходят в emergency wall-climb;
3. направлением остаётся запрошенная целевая площадка;
4. жители ползут по стене шахты к цели;
5. восстановление питания не возвращает уже вышедших пассажиров в кабину.

Emergency action сохраняет исходный request/target и не выбирает ближайшую или исходную площадку заново.

## Ездовой хомяк и ховерборд: восстановленная логика scripts

Источники: `scripts/classes/zwerg/z_dignwalk.tcl`, `z_dignwalk.tcl_copy`, `scripts/classes/items/transport.tcl`, `scripts/classes/zwerg/z_anims.tcl`.

Подтверждённое поведение legacy:

- средство используется автоматически, если соответствующий tool находится в Inventory гнома;
- для обычного пути включается `speedtype 3`, помеченный в scripts как `autodist`: движок сам выбирает быстрое движение на достаточно длинном маршруте;
- для повторной/принудительной быстрой команды используется `speedtype 2`;
- ховерборд проверяется раньше ездового хомяка и имеет приоритет, если оба предмета находятся в Inventory;
- короткое перемещение может остаться обычной ходьбой;
- перенос BuildingBox (`-withbox`) запрещает fast movement;
- ховерборд использует собственный board animation set;
- ездовой хомяк использует собственный riding animation set;
- оба являются tool items, а не отдельными пассажирскими транспортными системами.

TCL не содержит отдельных числовых множителей скорости для этих двух предметов. Оба передают одинаковые `speedtype 2/3` во встроенную engine walk-action и отличаются только предметом/анимацией. Поэтому Dig сохраняет одинаковую базовую mobility policy для обоих, а конкретный speed multiplier относится к Q-014 `BALANCE_TBD` и задаётся в data definitions, не выводится из выдуманного legacy-числа.

## Владение состояния

Navigation владеет links/routes. Transport владеет elevator cabin, requests, passengers и mobility equipment state. Energy сообщает доступность лифта. Inventory владеет Reithamster/Hoverboard items. Jobs/Reservations владеет местами и очередью. Presentation показывает состояние, но не рассчитывает маршрут.

## Инварианты

- один passenger slot не используется дважды;
- автоматический лифт использует только валидные площадки;
- manual request может выбрать любую высоту шахты;
- emergency climb всегда сохраняет target destination;
- cargo существует только как personal Inventory;
- Hoverboard имеет deterministic priority над Reithamster;
- BuildingBox отключает fast mobility;
- неизвестные engine speed multipliers не копируются предположительно.

Q-051 закрыт. Числовая скорость ездового хомяка и ховерборда остаётся обычным balance-параметром Q-014, поскольку в TCL она отсутствует.
